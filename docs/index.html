DevOps DB Tool

Инструмент для работы со схемами PostgreSQL: экспорт RTF-документации, резервное копирование DDL и анализ качества комментариев к объектам базы данных.

Требования к системе

Операционная система

  • Linux (рекомендуется), macOS или Windows
  • Qt 5.15 или новее (входит в состав исполняемого файла либо устанавливается отдельно)

PostgreSQL-клиент

Для разделов «Описание схемы» и «Бэкап / Восстановление» на машине должны быть установлены клиентские утилиты PostgreSQL:

# Debian / Ubuntu
sudo apt install postgresql-client

# Fedora / RHEL
sudo dnf install postgresql

Убедитесь, что psql и pg_dump доступны в PATH:

psql --version
pg_dump --version

Доступ к базе данных

  • Пользователь должен иметь права CONNECT на целевую базу данных.
  • Для просмотра таблиц и комментариев достаточно роли pg_read_all_data или явных SELECT-привилегий.
  • Для создания резервных копий через pg_dump нужна роль pg_dump-совместимого пользователя (обычно postgres или суперпользователь).

Запуск программы

./devops_db_tool

При первом запуске поля подключения заполнены значениями по умолчанию (127.0.0.1:5432, пользователь postgres). Настройки автоматически сохраняются и восстанавливаются при следующем запуске.

✔ Соединение успешно ✗ Текст ошибки

Раздел «Описание схемы»

Генерирует RTF-документ с описанием всех таблиц и колонок указанной схемы PostgreSQL. В документ включаются: название таблицы, её комментарий, для каждой колонки — название, тип данных, комментарий и, опционально, внешний ключ.

Пошаговая инструкция

  1. Заполните параметры подключения и нажмите «Проверить соединение».
  2. Введите имя схемы (например, public или myschema).
  3. Укажите директорию вывода — папку, куда будет сохранён RTF-файл. Используйте кнопку «Обзор…» для выбора через диалог.
  4. При необходимости включите опцию «Включить описание Foreign Key».
  5. Нажмите «Сгенерировать описание схемы (RTF)».
  6. По окончании в поле «Последний сгенерированный файл» появится путь к RTF. Кнопка «Скопировать путь» скопирует его в буфер обмена.

Имя файла

Файл сохраняется в выбранной директории с именем вида:

schema_<имя_схемы>_description_<YYYYMMDD_HHmmss>.rtf

Структура RTF-документа

  • Заголовок схемы и её комментарий
  • Для каждой таблицы: заголовок, комментарий, таблица колонок
  • Колонки таблицы: Наименование поля / Описание (рус.) / Тип данных и опционально Foreign Key
Совет Лог в нижней части экрана показывает прогресс обработки каждой таблицы. Если таблица обрабатывается долго — возможно, схема содержит большой объём данных в системных каталогах.

Раздел «Бэкап / Восстановление»

Создаёт резервную копию DDL-структуры схемы в виде двух SQL-скриптов, а также позволяет применить эти скрипты к базе данных для восстановления.

Параметры подключения

Аналогичны разделу «Описание схемы» (host, порт, база данных, пользователь, пароль, схема).

Экспорт (создание резервной копии)

Почему два скрипта? При восстановлении схемы в пустую базу данных внешние ключи (FOREIGN KEY) могут мешать созданию таблиц, если целевая таблица ещё не существует. Разделение на два скрипта позволяет сначала создать все таблицы, а затем добавить FK-ограничения.
  1. Заполните параметры подключения, нажмите «Проверить соединение».
  2. Укажите директорию вывода через кнопку «Обзор…».
  3. Нажмите «Экспортировать (создать 2 скрипта)».
  4. После завершения в лог будут выведены пути к двум файлам, а поля «Шаг 1» и «Шаг 2» заполнятся автоматически.

Имена создаваемых файлов

schema_<схема>_nodeps_<YYYYMMDD_HHmmss>.sql   ← Скрипт 1: таблицы без FK
schema_<схема>_fk_only_<YYYYMMDD_HHmmss>.sql   ← Скрипт 2: только FK

Восстановление (импорт скриптов)

Важно: порядок применения Сначала всегда применяйте Скрипт 1 (структура без FK), и только после его успешного завершения — Скрипт 2 (FK-ограничения).
  1. Убедитесь, что параметры подключения указывают на целевую базу данных.
  2. В блоке «Шаг 1 — схема без FK» выберите файл …_nodeps_….sql через «Обзор…» или введите путь вручную.
  3. Нажмите «Применить к БД» рядом со скриптом 1 и дождитесь сообщения об успехе.
  4. В блоке «Шаг 2 — только FK» аналогично выберите файл …_fk_only_….sql и нажмите «Применить к БД».

Статус каждой операции и вывод psql отображаются в лог-панели внизу.

Раздел «Анализ комментариев»

Сканирует все пользовательские схемы базы данных и оценивает качество комментариев к таблицам и колонкам. Помогает выявить объекты без описания и комментарии с латиницей там, где ожидается только кириллица.

Параметры подключения

ПолеОписаниеПо умолчанию
HostАдрес сервера PostgreSQL127.0.0.1
ПортПорт сервера5432
База данных *Имя базы данных—
ПользовательИмя пользователяpostgres
ПарольПароль пользователя—

Схема не указывается — программа автоматически находит и анализирует все схемы базы данных (кроме системных: pg_catalog, information_schema).

Пошаговая инструкция

  1. Заполните параметры подключения и нажмите «Проверить соединение».
  2. Нажмите «Анализировать». Программа последовательно обходит все схемы — ход анализа виден в строке статистики.
  3. После завершения левая панель заполнится списком схем с цветовой индикацией.
  4. Кликните на любую схему — справа появится подробная статистика и список проблемных объектов.

Цветовая индикация схем

✓ Нет проблем ⚠ Есть замечания ✗ Серьёзные проблемы
ЦветУсловие
✓ Зелёный Все таблицы и колонки имеют комментарии, латиница отсутствует.
⚠ Жёлтый Покрытие комментариями ≥ 80 % для таблиц и колонок, но есть отдельные замечания.
✗ Красный Покрытие комментариями < 80 % хотя бы для таблиц или колонок.

Панель статистики

При выборе схемы в правой части отображается:

  • Комментарий к самой схеме (есть / пустой)
  • По таблицам: всего / с комментарием / только кириллица / есть латиница / без комментария
  • По колонкам: аналогичные показатели

Фильтр проблемных объектов

Выпадающий список позволяет переключаться между категориями проблем (в скобках — количество):

  • Таблицы без комментария — таблицы, у которых комментарий не заполнен.
  • Таблицы с латинскими буквами — комментарии содержат латиницу.
  • Колонки без комментария — колонки без описания.
  • Колонки с латинскими буквами — комментарии к колонкам содержат латиницу.
Автовыбор После анализа автоматически выделяется первая «красная» схема, чтобы сразу показать наиболее проблемное место.

Частые вопросы

Программа сообщает «Ошибка соединения» при проверке

Убедитесь, что:

  • сервер PostgreSQL запущен и доступен по сети;
  • в pg_hba.conf разрешено подключение с вашего IP;
  • логин и пароль введены верно;
  • указана существующая база данных.

RTF-файл открывается, но текст отображается некорректно

Файл создаётся в кодировке Windows-1251 (cp1251). Откройте его в Microsoft Word, LibreOffice Writer или любом RTF-совместимом редакторе. Если используете текстовый редактор — убедитесь, что он распознаёт кодировку.

pg_dump завершается с ошибкой при экспорте

  • Проверьте, что pg_dump установлен и доступен в PATH.
  • Версия pg_dump должна совпадать с основной версией сервера PostgreSQL.
  • Пользователь должен иметь право читать схему: роль pg_read_all_data или явные права.

При применении скрипта 1 возникает ошибка «already exists»

Схема или её объекты уже существуют в целевой базе. Либо удалите существующую схему вручную (DROP SCHEMA … CASCADE), либо используйте новую пустую базу данных.

Анализ комментариев занимает много времени

Время анализа пропорционально количеству схем и объектов в базе данных. Для баз с тысячами таблиц анализ может занимать несколько минут — следите за прогрессом в строке статистики.

Обратная связь

Обнаружили ошибку или хотите предложить новую функцию? Напишите на электронный адрес, указанный в разделе «О программе».

При описании ошибки, пожалуйста, укажите:

  • версию программы (из раздела «О программе»);
  • версию PostgreSQL (SELECT version(););
  • операционную систему;
  • точное описание шагов для воспроизведения ошибки;
  • текст сообщения об ошибке или содержимое лог-панели.
