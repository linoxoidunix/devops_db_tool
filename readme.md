# DevOps DB Tool

Утилита для работы со схемой PostgreSQL: генерация RTF-описания и бэкап/восстановление схемы.

## Зависимости

| Компонент | Версия |
|-----------|--------|
| Qt        | ≥ 5.15 |
| CMake     | ≥ 3.16 |
| C++       | 17     |
| psql      | в PATH |
| pg_dump   | в PATH |

## Сборка

```bash
cmake -B build
cmake --build build
./build/devops_db_tool
```

## Интерфейс

При запуске открывается окно с двумя вкладками:

```
┌─────────────────────────────────────────────────────────────┐
│  DevOps DB Tool                                             │
├──────────────────────┬──────────────────────────────────────┤
│  Описание схемы      │  Бэкап / Восстановление              │
└──────────────────────┴──────────────────────────────────────┘
```

## Вкладка «Описание схемы»

Подключается к PostgreSQL и генерирует RTF-файл с описанием всех таблиц и колонок заданной схемы.

```
┌─ Подключение к БД ──────────────────────────────────────────┐
│  Host:           [ 127.0.0.1          ]  Порт: [ 5432 ]     │
│  База данных:    [ my_database        ]                      │
│  Пользователь:   [ postgres           ]                      │
│  Пароль:         [ ••••••••           ]                      │
│  Схема:          [ public             ]                      │
└─────────────────────────────────────────────────────────────┘
┌─ Директория вывода ─────────────────────────────────────────┐
│  [ /home/user/reports    ]  [Обзор…]  [Скопировать путь]    │
└─────────────────────────────────────────────────────────────┘
┌─ Опции отчёта ──────────────────────────────────────────────┐
│  ☐  Включить описание Foreign Key в колонках                │
└─────────────────────────────────────────────────────────────┘
│         [ Сгенерировать описание схемы (RTF) ]              │
┌─ Последний сгенерированный файл ────────────────────────────┐
│  [ /home/user/reports/schema_public_description_….rtf ]     │
│                                            [Скопировать путь]│
└─────────────────────────────────────────────────────────────┘
┌─ Лог ───────────────────────────────────────────────────────┐
│  Генерация RTF-описания схемы «public»…                     │
│    Схема: public                                            │
│    Таблиц найдено: 42                                       │
│    Обработка: users                                         │
│    Обработка: orders                                        │
│  ✔ RTF-отчёт сохранён: /home/user/reports/schema_public_…  │
└─────────────────────────────────────────────────────────────┘
```

### Как использовать

1. Заполните параметры подключения к БД.
2. Укажите директорию, куда сохранить RTF-файл (кнопка **Обзор…**).
3. При необходимости отметьте **«Включить описание Foreign Key»**.
4. Нажмите **«Сгенерировать описание схемы (RTF)»**.
5. Путь к готовому файлу появится в поле «Последний сгенерированный файл» — его можно скопировать одной кнопкой.

### Что содержит RTF-отчёт

Для каждой таблицы схемы формируется раздел:

```
Схема: public
  (комментарий к схеме из pg_namespace)

Таблица: users
  (комментарий к таблице)

┌──────────────────────┬──────────────────────┬────────────┐
│ Наименование поля    │ Описание (рус.)       │ Тип данных │
├──────────────────────┼──────────────────────┼────────────┤
│ id                   │ Идентификатор         │ integer    │
│ email                │ Электронная почта     │ varchar(255│
│ created_at           │ Дата создания         │ timestamp  │
└──────────────────────┴──────────────────────┴────────────┘
```

> Описания берутся из `COMMENT ON COLUMN` — убедитесь, что комментарии проставлены в схеме.

Файл именуется: `schema_{схема}_description_{YYYYMMDD_HHmmss}.rtf`

## Вкладка «Бэкап / Восстановление»

Экспортирует схему через `pg_dump` и разбивает результат на два SQL-скрипта. Те же скрипты можно применить к другой БД для восстановления.

```
┌─ Подключение к БД ──────────────────────────────────────────┐
│  Host:           [ 127.0.0.1          ]  Порт: [ 5432 ]     │
│  База данных:    [ my_database        ]                      │
│  Пользователь:   [ postgres           ]                      │
│  Пароль:         [ ••••••••           ]                      │
│  Схема:          [ public             ]                      │
└─────────────────────────────────────────────────────────────┘
┌─ Экспорт схемы (2 скрипта) ─────────────────────────────────┐
│  Директория: [ /home/user/backups ]  [Обзор…]               │
│                                                              │
│  Скрипт 1: схема без FK-ограничений                         │
│  Скрипт 2: только FK-ограничения (ALTER TABLE … FOREIGN KEY)│
│                                                              │
│         [ Экспортировать (создать 2 скрипта) ]              │
└─────────────────────────────────────────────────────────────┘
┌─ Импорт схемы ──────────────────────────────────────────────┐
│  Шаг 1 — схема без FK:                                      │
│  [ schema_public_nodeps_….sql   ]  [Обзор…]  [Применить к БД]│
│                                                              │
│  Шаг 2 — только FK:                                         │
│  [ schema_public_fk_only_….sql  ]  [Обзор…]  [Применить к БД]│
└─────────────────────────────────────────────────────────────┘
┌─ Лог ───────────────────────────────────────────────────────┐
│  Запуск pg_dump для схемы «public»…                         │
│    pg_dump завершён, разбиваем на 2 скрипта…               │
│  ✔ Скрипт 1 (без FK, 198 блоков): …/schema_public_nodeps_… │
│  ✔ Скрипт 2 (только FK, 34 ограничения): …/schema_public_… │
└─────────────────────────────────────────────────────────────┘
```

### Почему два скрипта

При восстановлении схемы в пустую БД FK-ограничения нужно добавлять **после** создания всех таблиц — иначе `psql` упадёт с ошибкой «referenced table does not exist».

| Скрипт | Содержимое | Имя файла |
|--------|-----------|-----------|
| 1 | `CREATE TABLE`, `CREATE INDEX`, `CREATE SEQUENCE`, `CREATE TYPE`, … | `schema_{схема}_nodeps_{ts}.sql` |
| 2 | `ALTER TABLE … ADD CONSTRAINT … FOREIGN KEY` | `schema_{схема}_fk_only_{ts}.sql` |

### Сценарий: бэкап

1. Заполните параметры подключения к **исходной** БД.
2. Укажите директорию для сохранения скриптов.
3. Нажмите **«Экспортировать (создать 2 скрипта)»**.
4. В логе появятся пути к двум SQL-файлам.

### Сценарий: восстановление на новой БД

> Целевая БД должна существовать и быть пустой (или не содержать конфликтующих объектов).

1. Переключите параметры подключения на **целевую** БД.
2. В поле «Шаг 1» укажите `schema_…_nodeps_….sql` (кнопка **Обзор…**).
3. Нажмите **«Применить к БД»** рядом с шагом 1 — создадутся таблицы без FK.
4. В поле «Шаг 2» укажите `schema_…_fk_only_….sql`.
5. Нажмите **«Применить к БД»** рядом с шагом 2 — добавятся FK-ограничения.

> После экспорта поля «Шаг 1» и «Шаг 2» заполняются автоматически.
